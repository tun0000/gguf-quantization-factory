# 實驗:量化品質崩潰的臨界點在哪?

## 動機

主表([summary-qwen2.5-3b-instruct.md](summary-qwen2.5-3b-instruct.md))只涵蓋 4-bit 以上
(Q4_K_M~F16),PPL 損失都在 +6% 以內,看不出「量化多激進會開始壞掉」。這個實驗把
Qwen2.5-3B-Instruct 一路量到 IQ3、IQ2 家族(3.7~2.0 bpw),找出真正的崩潰臨界點。

## 方法

- 用既有的英文 wikitext-2 imatrix(`imatrix.dat`)校正 IQ3/IQ2 系列(llama.cpp 官方建議
  IQ2/IQ3 一定要搭 imatrix,否則品質會更差)
- 8 個新等級:IQ3_M / IQ3_S / IQ3_XS / IQ3_XXS(3.66→3.06 bpw)、
  IQ2_M / IQ2_S / IQ2_XS / IQ2_XXS(2.7→2.06 bpw)
- 評測方式與主表一致:wikitext-2-raw-v1 test split、`llama-perplexity -c 512 -ngl 99`
- **VRAM 說明**:IQ2 那批量測時,Windows 側的 Ollama 剛好有其他模型(`qwen3:8b` 等)
  常駐占用 ~9.3GB,把 nvidia-smi 峰值墊高了。下表改用 `measure_vram.sh` 記錄的
  **delta**(量測前後的淨增量),不受背景佔用影響,可跨等級公平比較。

## 崩潰曲線的形狀:兩段斷崖,不是漸進滑坡

1. **第一段斷崖(4-bit → 3-bit)**:+3.4% → +10~11%,損失約跳 3 倍。3-bit 家族內部
   (IQ3_M/S/XS 三者互相在誤差範圍內,差異不顯著)品質相對平坦,直到 IQ3_XXS 才明顯
   再往下掉一階(+16.36%)——這是 3-bit 內部的次要斷崖。
2. **第二段斷崖(3-bit → 2-bit)**:IQ3_XXS(+16.36%)→ IQ2_M(+38.36%),損失再跳超過
   2 倍。這才是真正的懸崖——**進入 2-bit 家族後,每降一級 PPL 就急遽惡化**,
   到 IQ2_XXS 時 PPL 已經是 F16 的 2.64 倍(+164%),數值上代表模型輸出品質
   已經實質崩潰(對照:Q4_K_M 只有 +5.6%)。
3. **速度/VRAM 幾乎不隨量化程度變化**:tg128 tok/s 在 IQ3~IQ2 全部維持在
   335~365 之間(bpw 越低理論頻寬需求越小,但差異被固定的 kernel/overhead
   蓋過);VRAM 增量也都落在 1.6~2.2GB 區間,說明對這個 3B 模型而言,
   激進量化省的是磁碟空間與載入時的頻寬,推論時的邊際 VRAM/速度效益不大。

## 結論

- **崩潰臨界點在 IQ2 家族**,具體來說是 **IQ2_M(+38%)就已經跨過「不建議一般使用」
  的門檻**,IQ2_XXS(+164%)是實質崩潰。
- **IQ3 家族仍算堪用**:雖然比 4-bit 明顯劣化(+10~16%),但還沒到崩潰程度,適合
  磁碟/頻寬極度受限、可以接受品質妥協的場景。
- 這條曲線是 Qwen2.5-**3B** 這個尺寸的結果;更大的模型(7B/14B+)通常對激進量化
  更耐受,崩潰點會往更低的 bpw 推移——這也是為什麼「用自己的 pipeline 對目標模型
  實測」比套用別人的經驗法則更可靠。
